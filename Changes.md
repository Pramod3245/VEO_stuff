Here's my complete analysis and the specific changes needed across all four files to get seamless, photorealistic split-scene video generation.

The core problem is that your agents generate scenes *independently* — there's no **shared anchor state** passed between them. When you stitch the clips, the model has no memory of what the previous scene looked like, so characters drift, lighting jumps, and motion continuity breaks. ## `Script_generation.py` — the biggest overhaul needed

**Problem:** `character_prompt` is a vague description. The model gets a new "statistical best guess" for the character each scene. `first_frame_image` and `last_frame_image` are generated by the LLM with no shared coordinate anchor.

**Changes:**

1. **Upgrade `ScriptOutput` to include a Character Bible**, not a summary. Force the system prompt to extract 15+ locked attributes — exact eye color/shape, jawline, a distinctive mark (scar, mole), hair texture/length, skin undertone, clothing detail per scene. These are `unchangeable_features` that get prepended verbatim into every scene's `characters` field.

2. **Add a `SceneAnchor` model** that carries forward:
```python
class SceneAnchor(BaseModel):
    last_frame_prompt: str   # exact static image prompt of the final frame
    last_frame_uri: str | None = None  # GCS URI after Imagen generates it
    seed: int | None = None
    lighting_descriptor: str  # e.g. "stadium floodlights, 5600K, hard shadows right"
    camera_position: str      # e.g. "low-angle, 35mm, subject fills lower 60%"
    motion_vector: str        # e.g. "subject's right arm mid-rise at frame end"
```

3. **Change `generate_script` to return anchors** alongside the scene list:
```python
def generate_script(...) -> tuple[list, list[SceneAnchor]]:
```
The first scene's anchor is bootstrapped from `first_frame_image`. Each subsequent scene's `first_frame_image` must *verbatim copy* the previous scene's `last_frame_prompt`.

4. **Upgrade the system prompt** to use the 8-part framework per the research doc — `Cinematography → Subject → Action → Environment → Atmosphere → Style → Audio → Format`. Hardcode `"24fps, 35mm or 85mm lens feel"` as a mandatory line.

## `veo_prompt.py` — add DP vocabulary and anchor injection

**Problem:** The `VeoPrompt` class has a typo (`veo_prompt=` instead of `description=` in `Field`). More critically, prompts use narrative language instead of cinematographic directives.

**Changes:**

1. **Fix the Field typo** — `veo_prompt.py` line 11 uses `veo_prompt="""` as a kwarg on `Field`, which silently does nothing. Change to `description="""`.

2. **Inject the anchor at call time.** Change `generate_veo_prompt` to accept an anchor:
```python
def generate_veo_prompt(scene_description: str, anchor: SceneAnchor | None = None) -> VeoPrompt:
    anchor_block = ""
    if anchor:
        anchor_block = f"""
CONTINUITY ANCHOR (MANDATORY — first 0.5s must resolve this):
- Incoming motion: {anchor.motion_vector}
- Incoming camera: {anchor.camera_position}
- Incoming lighting: {anchor.lighting_descriptor}
- First frame must match: {anchor.last_frame_prompt}
"""
    user_prompt = anchor_block + scene_description
```

3. **Strengthen the system prompt** with DP-vocabulary instructions. Add a section:
```
CINEMATOGRAPHY LAYER (MANDATORY FRONT-LOADING):
Always begin the prompt with the camera body + lens feel:
  "Handheld Arri Alexa footage, 35mm anamorphic lens, 24fps"
This sets the texture profile before any subject description.
```

4. **Add physics-rich directives** for personal care / fluid scenes:
```
FLUID PHYSICS (for scenes with water, foam, sweat):
Do NOT write "sweaty". Write:
  "viscous beads breaking from skin surface, high-specularity damp sheen 
   on brow reflecting stadium floodlights"
For lather: describe bubble microscopics, water carving paths through foam,
skin revealed beneath as water streams.
```

## `Edit_scene.py` — add momentum inheritance

**Problem:** The editor is told to "modify only what was requested" — which prevents it from correcting lighting, motion, or camera position errors from the original that would cause stitching glitches.

**Changes:**

1. **Pass the anchor into the edit agent:**
```python
def edit_scene(user_prompt: str, anchor: SceneAnchor | None = None) -> EditScene:
    context = f"CONTINUITY ANCHOR:\n{anchor.model_dump_json()}\n\n" if anchor else ""
    full_prompt = context + user_prompt
```

2. **Add a Motion Continuity Rule to the system prompt:**
```
MOTION CONTINUITY RULE:
If an anchor is provided, the first 0.5s of your scene_description MUST
physically resolve the incoming motion_vector — do not start from rest.
Example: if anchor shows "right arm mid-rise", your 0.0s must show that
arm continuing upward before any new action begins.
```

3. **Add a Lighting Anchor Rule:**
```
LIGHTING ANCHOR RULE:
The lighting mood described in the anchor's lighting_descriptor must 
appear VERBATIM in your 0.0s–1.0s timestamp segment.
You may transition lighting after 1.0s if the user's edit requires it.
```

def edit_full_script(user_prompt: str, previous_script: dict, anchors: list[SceneAnchor]) -> ScriptOutput:

        f"Scene {i+1} anchor: {a.model_dump_json()}"

3. **Update system prompt** with neighbor-awareness:
```
NEIGHBOR CONTINUITY:
When modifying scene N, you MUST check the anchor for scene N-1.
The first timestamp of scene N must resolve scene N-1's motion_vector.
When modifying scene N, you MUST update scene N+1's first timestamp
to resolve scene N's new ending state.
Never edit only one scene in isolation — always check its neighbors.
```

## Orchestration layer (new) — the glue you're missing

The research doc's most important point is **First-and-Last Frame Anchoring**. You need a thin orchestrator that threads anchors across the pipeline:

```python
async def generate_full_ad(prompt: str, n_scenes: int, max_sec: int):
    scenes, initial_anchors = generate_script(prompt, n_scenes, max_sec)
    
    final_clips = []
    anchors = initial_anchors
    
    for i, scene in enumerate(scenes):
        anchor = anchors[i - 1] if i > 0 else None
        
        # 1. Generate first/last frame images via Imagen
        if anchor:
            scene["first_frame_uri"] = anchor.last_frame_uri  # ← hard link
        
        # 2. Generate Veo prompt with anchor injection
        veo_prompt = generate_veo_prompt(scene["description"], anchor)
        
        # 3. Call Veo with first_frame_uri (Frames-to-Video mode)
        clip_uri = await call_veo(
            prompt=veo_prompt.veo_prompt,
            negative_prompt=veo_prompt.veo_negative_prompt,
            first_frame_uri=scene.get("first_frame_uri"),
            seed=anchor.seed if anchor else None
        )
        
        # 4. Extract last frame, update anchor for next scene
        last_frame_uri = await extract_last_frame(clip_uri)
        anchors[i] = SceneAnchor(
            last_frame_prompt=scene["last_frame_image"],
            last_frame_uri=last_frame_uri,
            seed=get_seed_from_generation(),   # log and reuse
            lighting_descriptor=extract_lighting(scene["description"]),
            camera_position=extract_camera(scene["description"]),
            motion_vector=extract_motion_end(scene["description"])
        )
        
        final_clips.append(clip_uri)
    
    return final_clips, anchors
```

The seed logging point is worth emphasizing — every successful Veo generation should log its seed. Reusing the same seed across scenes biases the diffusion process toward the same stylistic patterns even when the prompt context changes. That alone will reduce color banding and lighting jumps significantly.
